1. The Problem: "Double Building" & Wrong Folder Structure
There were two main issues colliding:

Jekyll Interference: GitHub Pages uses a tool called Jekyll by default to build websites. Jekyll ignores any folders that start with an underscore (like _static or _images), which are critical for your book's styling. It also tries to process Markdown files itself. Because it didn't understand the special "admonition" syntax used by your book software (Sphinx), it just displayed the raw text.
Messy Branch History: Your gh-pages branch likely contained a copy of your entire project (including source files and the _build folder), rather than just the final HTML. This meant GitHub was trying to serve the source code instead of the result.
2. The Solution: The "Clean Slate" Approach
I performed a "Force Push" to completely overwrite the history of your website branch. Here is the step-by-step of what I did in the terminal:

cd .../_build/html: I went directly into the folder where your finished HTML files live.
rm -r .git: I deleted the hidden Git history folder inside that build directory. This made the folder "forget" it was part of a larger project, effectively treating it as a brand new, separate website.
git init & git checkout -b gh-pages: I started a fresh Git repository right there and named the branch gh-pages.
git add .: I staged all the files. Because I was inside _build/html, the 
index.html
 and 
intro.html
 became the root files, not buried inside a subfolder.
git push -f origin gh-pages: The -f (force) flag is the key. It told GitHub: "Ignore everything you currently have on this branch. Delete it all and replace it exactly with these files."
Summary
We stopped GitHub from trying to "build" your book and instead just forced it to host the static HTML files you had already successfully built on your computer.
